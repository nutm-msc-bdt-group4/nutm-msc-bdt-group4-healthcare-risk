🛠 Team Workflow &\ Standards



1\. Branching Strategy

• Main: Stable, submission-ready code only.

• Develop: The primary integration branch where all features meet.

• Feature: Individual tasks (Format: feature/name-task). 

◦ Example: feature/adeiza-spark-config or feature/precious-data-cleaning.



2\. The "Law of the PR"

1\. No Direct Pushes: Never push directly to main or develop.

2\. Feature First: All work must happen in a dedicated Feature Branch.

3\. Review: Open a Pull Request (PR) on GitHub. At least one other teammate must review/comment before merging into develop.



3\. Professional Commit Messages

We use Conventional Commits (type: description) to keep our audit trail clean for our lecturer's supervision:

• feature: A new feature or script (e.g., feature: add heart-rate visualization)

• fix: A bug fix (e.g., fix: resolve null values in admissions table)

• docs: Documentation only (e.g., docs: update setup instructions)

• style: Formatting changes that don't affect code logic.

• refactor: Code changes that improve performance without changing functionality.
